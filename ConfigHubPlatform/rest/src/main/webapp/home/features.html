Hierarchical Context Properties

All running applications have a context. It may be as simple as "Development", or wider scoped depending on the size of the organization. Generally, for each application context there is a set of corresponding configuration files.

Many times, these configuration files have same key-value pairs defined across applications in different contexts. This makes sweeping changes, even for a single key-value pair both difficult and risky.

To address this issue, ConfigHub assigns context to each key-value pair instead. We call these properties, context-properties. The result of this strategy is that property with the same context is only defined once, regardless of the number of applications. Changing a context-property value becomes simple and risk-free.

Context Files

Upload configuration files into your ConfigHub repository, and pull them into your application on request through the API. The notation ${ property_key }, lets you specify a property key, which will be replaced by its resolved value at the time of the request.

The editor gives you a side by side view of file content and the keys/values used for specific context. The simplicity of context files saves time and code, and removes the necessity for the application to manage environmental properties.

Security

At the heart of ConfigHub's security mechanism are the Security Groups. They are password protected, and accept as members both property keys and files. Optionally, a security group can enable encryption using specified cypher, and if enabled, it will store encrypted all assigned files and key's values. From a UI standpoint, a security group password-protects a resource (key's values and/or files) from editing, and if encryption is enabled, hide the data from being visible.

Pulling Security-Group Encrypted Data

There are several ways for an application to pull properties and/or files that are encrypted using one of the Security Profiles. Which strategy, or combination of strategies, you end up using will largely depend on your security requirements.

Pre-Authorized Token

Security Groups with encryption enabled can be added to any token, and thus pre-authorize the token to decrypt all data encrypted by that security group.

Using this strategy, a client does not need to have any knowledge about the security surrounding this data.

Password on Request

During a pull request, a client may specify a password for any security group. Any data returned that belongs to these security groups will be decrypted before it is returned.

API Documentation

Return Encrypted

By default, if neither decryption strategy is used, data will be returned encrypted. With the encrypted data, information cipher and security group name is returned.

All ConfigHub transactions are encrypted using strong 2048-Bit SSL Certificate Encryption.

Revisions

Revisions

Changes made to any and all components of your repositories are recorded. Each change, a revision, is categorized and easily recalled so that you always have visibility into your configuration history.

Either from the API, or configuration editor, you may specify a point in time and recall your repository exactly as it was at that time. This allows you to compare configuration from any point in time, to any other point in time. It also allows you to pull configuration via APIs for a specific date/time, or a pre-defined tag.

Teams / Access Controls

Roles of those that have access to application configuration is not the same. You may only want to allow development config to be managed by developers. Or even, multiple product suites may have entirely different groups. Managed access gives you full control over who has access to which config resource.

Type-Checked Values

When your code expects a specific value type (boolean, int, long, etc.) ConfigHub may enforce this per key, though a Value-Type key attribute.

We currently support following Value-Types: String, Boolean, Integer, Long, Double, Float, Map, List, and Code/ML - code or markup syntax highlighting.

In the near-term you can expect Value-Type template support where you can add your own value type validation for custom types.

Tags / Versions

Whether you are cutting a release, or just want to leave a reference point with a note for a specific point in time of your repository, tags let you do that. They are easily recalled when editing or comparing configuration, or when pulling configuration through the API.

Context / Time Comparison

Being able to see the configuration difference between contexts is critical in early error detection. Unfortunately, due to the complexities of configuration files, it is often too complex to do this using only file-diffing tools.

ConfigHub let's you compare any context to any other context, allowing you full range of complex comparisons, including comparisons over time. So, comparing environment, application, or any other context permutation is as easy as specifying them.

Organizations

Organizations are a management groups that allow you to define owners and administrators across all your repositories. It is another way to share and control responsibility of managing contributors, access controls, modification and deletion of critical repository components.
